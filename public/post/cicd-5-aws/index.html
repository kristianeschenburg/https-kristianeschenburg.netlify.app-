CI/CD Part 4: Container Registries

Building images and pushing to container registries

Jun 15, 2023 2 min read

Building a package locally is straightforward, but doing so within a Gitlab CI/CD pipeline is a little more complicated. But, we can imagine adding this type of job to each of our repositories, and conditioning it on a merge request (or something of that kind).

If you havn’t yet, take a look at the CI/CD Authentication notes. That’ll explain the setup_tokens.sh script that you’ll use in this YAML snippet, as well as the definition of the .pypirc file that is necessary for pushing to a package registry.

I’ve called this job build-package and it belongs to a stage also called build-package. We first set up the .pypirc and .netrc files in the Docker image doing the package building. We then install the build and twine libraries in the before_script attribute. In order for build to properly build your package, you need to set up a pyproject.toml file that describes your package – you can follow the instructions described here.

In the script attribute of the job, we’ll build our package, and push it to our package registry (again, we’ve defined the registry in our .pypirc file – here, it’s referred to as the “personal” registry.)

#
image: python:3.9-slim

stages:
  - build-package

#### BUILDING PACKAGE AND PUSHING TO GITLAB PACKAGE REGISTRY
build-package:
  stage: build-package
  before_script:
    # see previous post on Authentication tokens for what setup_tokens.sh does
    - chmod +x ./setup_tokens.sh; ./setup_tokens.sh
    - apt-get update
    - apt-get install --yes --no-install-recommends gcc g++ libffi-dev
    # build and twine installation
    - python3 -m pip install build twine
  script:
    # build package
    - python3 -m build
    # push package to registry
    - python3 -m twine upload --repository personal dist/* --verbose
